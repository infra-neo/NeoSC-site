Flujo de Compra End-to-End

Desde que el usuario elige el plan hasta que recibe el email con acceso

🛍️ Fase 1 — Selección y configuración

Usuario visita /market
Elige plan (Starter/Business/Enterprise), ajusta vCPU/RAM/disco con sliders, selecciona TSplus 5/10/25 licencias, activa addons, elige región.
POST /api/market/price → precio reactivo en tiempo real (público, sin auth)
Click "Continuar al pago"
Si no está autenticado → redirect a /login?from=/market → Zitadel OIDC PKCE → callback → token JWT → regresa a /market/checkout con la config guardada en sessionStorage.
Crear orden en backendPOST
POST /api/market/orders — backend recalcula el precio (nunca confía en el frontend), crea documento en market_orders, genera los 12 provision_steps en estado pending. Devuelve order_id.

💳 Fase 2 — Pago

Checkout — elige método de pago
Modo Demo: POST /api/market/orders/{id}/simulate-payment → marca orden paid → asyncio.create_task(_simulate_provisioning)
Stripe: PaymentIntent → webhook payment_intent.succeeded → marca paid → dispara Celery task
PayPal: redirect → capture → marca paid → dispara Celery task
Redirect a pantalla de progreso
Frontend navega a /market/progress?order_id=xxx y abre EventSource SSE al endpoint GET /api/market/orders/{id}/stream

✅ Fase 3 — Post-pago

Orquestador ejecuta 12 pasos
Cada paso actualiza provision_steps en MongoDB. El SSE stream (polling cada 2s) detecta cambios y emite data: {step, status, log} al frontend. Frontend anima los step cards en tiempo real.
VM lista → email + redirect
Step complete → frontend muestra panel de éxito con URL de acceso → redirect automático a /workspaces donde la nueva VM aparece en la lista junto con las de Emergent.

Orquestador — 12 Pasos de Provisioning

Cada paso actualiza MongoDB + emite evento SSE al frontend en tiempo real

# Step Duración demo Servicio invocado Qué hace Fallo = ?
0 payment_confirmed 0.5s MongoDB Marca pago verificado. No hace nada más — ya está validado antes de llegar aquí. —
1 generate_credentials 2s TSplus API + Netbird API Genera licencia TSplus perpetua (TSPLUS-XXXX). Crea setup key Netbird para el peer. Guarda ambas en market_orders. Orden queda en failed. No se creó nada en infra.
2 create_lxd_vm 8s apache-libcloud → LXD Crea instancia Windows en bare-metal via apache-libcloud (driver intercambiable: LXD / GCP / AWS). Espera hasta que arranca. Registra market_vms con IP interna. VM no existe. Reintento limpio posible.
3 windows_bootstrap 15s Ansible (WinRM) Playbook windows_base.yml: WinRM HTTPS, OpenSSH + llave SSH soporte, RDP solo desde CGNAT Netbird (100.64.0.0/10), High Perf power plan, múltiples sesiones RDP, NLA desactivado para TSplus. VM existe pero no configurable. Retry WinRM.
4 tsplus_install 20s Ansible (WinRM) Playbook tsplus_install.yml: descarga instalador, instala silent, activa licencia perpetua con key del paso 1, activa support license, habilita HTML5 portal en puerto 80. TSplus no activo. Retry install.
5 netbird_install 8s Ansible (WinRM) Playbook netbird_install.yml: instala Netbird silent con setup_key del paso 1, conecta al management (manager.kappa4.com), configura autostart + watchdog cada 5 min. Registra IP Netbird (100.x). VM no en mesh. Acceso fallback por IP interna.
6 tsplus_configure 5s WinRM directo Vía AdminTool CLI: max sesiones = licencias contratadas, habilita HTML5 gateway, configura nombre de empresa, publica escritorio remoto como app, reinicia Apache TSplus. HTML5 no disponible. Retry AdminTool.
7 netbird_configure 5s Netbird API Crea grupo tenant-{id[:8]}, agrega peer al grupo, crea policy ACL (solo peers del mismo grupo se comunican), crea DNS interno {vm_name}.nb.kappa4.com. Acceso no restringido. Warning, no crítico.
8 zitadel_provision 6s Zitadel API Crea organización Zitadel para el tenant, proyecto WinDesk, roles admin/user, usuario admin con contraseña temporal (force change), apps OIDC para Netbird y para el portal web, Action policy max 2 dispositivos/usuario. Sin SSO propio. Usa SSO global.
9 dns_create 2s Cloudflare API Crea registro A en zona kappa4.com: {hostname}.desk.kappa4.com → IP Netbird. TTL 120s. Cloudflare proxy desactivado (WireGuard necesita IP directa). Acceso solo por IP. DNS puede retry.
10 email_welcome 1s SendGrid / SES Envía email al tenant con: URL de acceso HTML5, IP Netbird, contraseña temporal Zitadel (force change), instrucciones MFA, link a documentación. No bloquea si falla. No bloquea. Warning en log.
11 complete ✓ 0.5s MongoDB Marca orden status=active, VM status=running, provisioned_at. SSE emite evento con datos de VM. Frontend muestra panel de éxito y redirige a /workspaces en 4s. —

Auth & Zero Trust — Capas de seguridad

Múltiples capas: Zitadel OIDC PKCE → Bearer JWT → Netbird ACL → TSplus max_sessions

👤 Zitadel OIDC PKCE — flujo completo

Usuario click "Sign in"
Frontend genera code_verifier + code_challenge (SHA256 base64url). Redirect a beyondcloud-nxm7ab.us1.zitadel.cloud/oauth/v2/authorize con PKCE params.
Zitadel autentica
Usuario ingresa credenciales → TOTP/WebAuthn MFA → Zitadel verifica → redirect a /auth/callback?code=xxx
Token exchangePOST
POST /api/auth/sso → backend intercambia code + code_verifier por access_token + id_token. Extrae claims: sub (user ID), roles, groups, org_id. Crea/actualiza user en MongoDB. Devuelve token al frontend.
Todas las requests autenticadas
Frontend envía Authorization: Bearer {token}. FastAPI valida con active_tokens dict (prod: Redis + JWKS). Grupos determinan permisos: admin → /admin, user → /workspaces, /market.

🏢 Zitadel por tenant (post-compra)

El orquestador crea vía API (paso 8):
Organización: NombreEmpresa
Proyecto: WinDesk Cloud
Roles: admin, user
App OIDC 1: Netbird (PKCE)
App OIDC 2: Portal Web (PKCE + secret)
Action: max 2 dispositivos/usuario
Usuario admin: {email} + temp password
El cliente hace login → Zitadel fuerza cambio de contraseña → puede registrar MFA → access a su portal dedicado.

Microservicios & Componentes

Cada servicio tiene responsabilidad única. Se comunican vía Celery queue o API directa.

🖥️ Frontend (React CRA)

Stack: React 19 · react-router-dom v7 · Tailwind · shadcn/ui · axios
Estado: useState local + sessionStorage para config market
Auth: AuthContext → Zitadel PKCE · localStorage tokens
SSE: EventSource nativo · polling fallback cada 3s
Precios: cálculo reactivo local + validación POST /price

⚡ API Gateway (FastAPI)

Stack: FastAPI 0.110 · Python 3.11 · Motor (MongoDB async)
Auth: Bearer token → active_tokens dict (→ Redis prod)
Routers: auth, market, workspaces, sessions, admin, audit
SSE: StreamingResponse con asyncio.sleep(2) polling loop
CORS: configurado via env CORS_ORIGINS

⚙️ Provision Worker (Celery)

Queue: provision (concurrency=3)
Infra: apache-libcloud (LXD/GCP/AWS) · pylxd fallback
Actual: asyncio.create_task (demo) · Celery en producción
Retry: max_retries=1, ack_late=True
Log: cada step → MongoDB provision_steps

🔧 WinRM Worker (Celery)

Queue: winrm (concurrency=2, serialized por VM)
Stack: ansible_runner · pywinrm
Playbooks: windows_base.yml · tsplus_install.yml · netbird_install.yml
Inventory: dinámico por VM (IP + creds WinRM)

🌐 Netbird Service

API: manager.kappa4.com/api · Token auth
Operaciones: create setup_key · create group · add peer · create policy · create DNS nameserver
Por tenant: 1 setup_key · 1 group · 1 ACL policy · DNS interno

👤 Zitadel Service

API: beyondcloud-nxm7ab.us1.zitadel.cloud
PAT: service account token para crear orgs
Por tenant: 1 org · 1 proyecto · 2 roles · 2 apps OIDC · 1 admin user · 1 Action
Action: max 2 dispositivos por usuario (post-session)

🌐 Cloudflare Service

API: api.cloudflare.com · Zone API token
Operación: POST /dns_records · Tipo A
Resultado: {hostname}.desk.kappa4.com → IP Netbird
TTL: 120s · Proxy: false (WireGuard necesita IP directa)

📦 TSplus Service

Install: silent /SILENT /SUPPRESSMSGBOXES
Activación: AdminTool.exe /activate {key} /silent
Config: setmaxsessions · enablehtml5 · enablewebaccess
DEMO: simulado en _simulate_provisioning()
PROD: requiere cuenta reseller TSplus API

📧 Notify Service

Provider: SendGrid / AWS SES (configurable)
Email bienvenida: URL acceso HTML5 · IP Netbird · contraseña temporal Zitadel · instrucciones MFA
No bloquea: fallo de email no detiene el provisioning

Flujo de datos durante provisioning

POST /orders → INSERT market_orders(pending) + INSERT 12 provision_steps(pending)
simulate-payment → UPDATE market_orders(paid) → asyncio.create_task()
cada step → UPDATE provision_steps(running) → delay → UPDATE(success)
SSE stream → SELECT provision_steps cada 2s → emite eventos al frontend
step complete → INSERT market_vms → UPDATE market_orders(active)
GET my-vms → JOIN market_orders(active) + market_vms → response cliente
